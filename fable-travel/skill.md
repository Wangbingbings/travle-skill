# fable-travel: 旅行研究

一条命令完成：三路并行研究 → 路线合成 → Markdown 文档 + 便携卡片。

## 执行策略

使用 Workflow 编排。step 2 通过 `parallel()` 并发 3 个 Agent，各自独立搜索，互不依赖。Workflow 主进程等待全部完成后进入路线合成。

## 参数

纯自然语言解析。用户怎么说都行——说城市名就行，其他信息可有可无，无需记任何 flag。

| 输入 | 类型 | 提取规则 | 示例 |
|------|------|---------|------|
| **城市名** | 必填 | 消息中第一个中文地名。如无法识别则反问用户。 | "西安"、"洛阳"、"大同" |
| **天数** | **必填** | 匹配数字+天/日/周末。用户未提及则反问"玩几天？"。 | "4天"、"周末两天"、"一日游" |
| **聚焦主题** | 可选 | 聚焦/主要/重点/关于/喜欢 等关键词后的内容。无聚焦则做全城综合研究。 | "聚焦唐代"、"主要看石窟"、"重点逛咖啡" |

**典型输入：**

| 你说... | 技能理解 |
|---------|---------|
| `帮我做西安的旅行研究，4天，聚焦唐代` | 城市=西安，天数=4，聚焦=唐代 |
| `洛阳出发前功课，2天` | 城市=洛阳，天数=2，无聚焦 |
| `大同 city walk 研究，2天` | 城市=大同，天数=2，无聚焦 |
| `北京周末两天，主要看石窟` | 城市=北京，天数=2，聚焦=石窟 |
| `苏州` | 城市=苏州 → 反问"玩几天？"，无天数不继续 |

## 执行

### 1. 解析参数

从用户消息中用自然语言提取：
- **城市名** — 必填，无城市名则要求用户补充
- **天数** — **必填**，用户未提及则反问"玩几天？"
- **聚焦主题** — 可选，如有则后续所有 Agent 的搜索围绕该主题展开

### 2. 三路并行研究

使用 Workflow 的 `parallel()` 同时运行以下 3 个 Agent。每个 Agent 接收的入参为 `{city, days, focus}`，返回结构化 Markdown。

**失败处理**：单个 Agent 失败不影响其他 Agent，失败 Agent 的结果以空结构替代，主进程继续执行。3 个 Agent 全部失败则终止并报错。

**Agent 1 — 博物馆 + 考古 + 历史（文化侧，占 ~70% 输出量）**

搜索策略：
- 百度搜索："{城市} 博物馆 镇馆之宝"、"{城市} 考古发现"、"{城市} 全国重点文物保护单位"
- 百度百科：城市历史沿革、重大考古发现
- 国家文物局/各省博物馆官网（如有）

量化目标：博物馆 3-6 个，考古发现 2-4 个，历史分层覆盖至少 3 个时期。

输出要求（结构化 markdown）：
```
## 博物馆
- {博物馆名} | 地址 | 预约方式
  - 镇馆之宝：{文物名} — 为什么重要 — 看什么细节
  - 重点展厅：{展厅名} — 核心看点
  - 容易错过的：{内容}

## 考古发现
- {发现名} — 意义 — 出土文物现藏地 — 有趣的发掘故事（如有）

## 历史分层
- {时期}（{年代}）— 核心事件 — 遗留痕迹 — 可看的实物
```

**Agent 2 — 新潮本地生活（潮流侧，占 ~30% 输出量）**

搜索策略：
- 小红书搜索："{城市} city walk"、"{城市} 小众"、"{城市} 独立咖啡"、"{城市} 市集"、"{城市} livehouse"、"{城市} 街头艺术"、"{城市} 本地人推荐"、"{城市} 年轻人"
- 如有聚焦主题："{城市} {主题} 小众"

量化目标：新潮地点 4-8 个，覆盖至少 3 种类型（咖啡/市集/livehouse/街区/展览）。

筛选标准：
- 要：有具体店名/地点、有地址、有真实体验描述
- 不要：纯打卡拍照、无信息量的"氛围感"帖、广告软文

输出要求（结构化 markdown）：
```
## 新潮地点
- {地点名} | 类型（咖啡/市集/livehouse/街区/展览/...）| 地址 | 所在区域
  - 为什么值得去：{一句话}
  - 怎么找到：{具体指引，如"XX路拐角，门口有棵大树"}
```

**Agent 3 — 探店 + 深度内容**

搜索策略：
- 小红书搜索："{城市} 探店"、"{城市} 本地美食"、"{城市} 老字号"、"{城市} 巷子里的吃的"、"{城市} 人均"、"{城市} 必吃"
- B站搜索："{城市} 博物馆 讲解"、"{城市} 考古"、"{城市} 深度"（筛选 10 分钟以上视频）

量化目标：探店 4-8 家，深度内容 2-5 条。

筛选标准：
- 探店：有具体地址、推荐菜、人均价格、真实评价
- B站视频：有知识增量，不是纯 vlog 打卡

输出要求（结构化 markdown）：
```
## 探店
- {店名} | 类型 | 地址 | 所在区域 | 人均
  - 推荐菜：{菜品}
  - 一句话：{为什么值得吃}
  - 和哪条路线搭配：{建议}

## 深度内容
- [{标题}]({URL}) — {一句话摘要} — 平台：{B站/公众号/知乎}
```

`parallel()` 等待 3 个 Agent 全部完成（失败者返回空数据结构）。

### 3. 路线合成

拿到 3 个 Agent 的结果后，主进程执行路线合成。路线是核心产出——地理位置聚合、交通效率优先，而非按主题分类。

**合成流程：**

1. **打地理标签** — 给所有地点标注所在区域（如"钟楼片区"、"曲江新区"、"回民街附近"、"老城根"）
2. **按区域聚合** — 地铁/步行 30 分钟内可达的地点归入同一路线（每条路线含 3-6 个地点）
3. **时间排序** — 每条路线内按地理位置顺序排列，不走回头路
4. **穿插探店** — 每条路线安排沿途餐饮（午餐/下午茶/晚餐），探店插入路线中间而非单独列出时间
5. **古建融入** — 如果某条路线经过古建/遗址，顺带写进路线说明，不设单独的古建章节

**冲突处理原则：**
- 两个推荐地点时间冲突 → 按"必去 > 可选"优先级筛选，同优先级则按地理顺路度取舍
- 同区域同类地点过多 → 保留评分最高/最有特色的 2-3 个
- 探店建议与路线方向不符 → 优先选择路线沿途的店，而非折返

**输出 schema（结构化 Markdown）：**

```
## 路线概览
共 {N} 条路线 | {最短}–{最长} 小时/条

## 路线一：{主题名}
适合：{场景描述，如"第一天上午，刚到"}
路线：{起点} → {途径点1} → {途径点2} → ... → {终点}
总时长：约 {N} 小时（含步行+参观+用餐）
交通方式：全程步行 / 地铁+步行
沿途餐饮：{午餐推荐} | {下午茶推荐} | {晚餐推荐}

### 详细行程
1. **{地点名}**（推荐停留 {N} 分钟）
   - 到了做什么：{操作式看点——不是"这是什么"，而是"到了先看XX，注意XX细节"}
   - 交通衔接：→ 沿{路名}向{方向}步行 {N} 分钟（经过{地标}）

2. **{地点名}**（推荐停留 {N} 分钟）
   ...

### 🍜 沿途用餐
- 午餐：{店名} | 推荐菜 | 人均 ¥{XX} | 位置：路线中段附近
- 下午茶：{店名} | 推荐 | 人均 ¥{XX}

---

## 路线二：{主题名}
...
```

**质量标准：**
- 每条路线必须有明确的起点和终点
- 每个地点必须有操作式看点（"到了做什么"，而非百科式介绍）
- 路线之间的交通衔接必须有具体指引（地铁站+出口+步行时长）
- 每条路线至少安排 1 个餐饮点

### 4. 合成 Markdown 文档

将研究结果和路线合成为一份结构化 Markdown 文档。

**文件命名与路径：**

```
C:\Users\{用户名}\OneDrive\Documents\travel-notes\{城市}旅行研究_{YYYY-MM-DD}.md
```

如果 OneDrive 路径不存在，回退到 `{用户文档目录}\travel-notes\`。

**文档结构：**

```markdown
# {城市}旅行研究

> {一句话——这座城市在干什么，年轻人为什么要去。不是百科介绍，是"去的理由"。}

## 路线

### 路线一：{主题名}（约{N}小时）
适合：{场景描述}
1. {地点} → 看什么 / 吃什么（停留建议时间）
2. ...
📍 交通：{地铁站} {出口} → 步行 {N} 分钟 | 或：{公交线路}

### 路线二：{主题名}
...

---

## 博物馆 + 考古

### {博物馆1名称}
- **地址**：{地址}
- **开放时间**：{时间} | **预约**：{方式}
- **镇馆之宝**：{文物名} — 为什么重要 — 看什么细节
- **重点展厅**：{展厅名} — 核心看点
- **容易错过的**：{内容}

### {博物馆2名称}
...

---

## 新潮本地生活

### {地点}（{类型}）
- **地址**：{地址}
- **为什么值得去**：{一句话}
- **怎么找到**：{具体指引}

---

## 探店

### {店名}（{类型}）
- **地址**：{地址} | **人均**：¥{XX}
- **推荐菜**：{菜品}
- **搭配路线**：路线{一/二/三}沿途

---

## 深度内容推荐

出发前值得看的：
- [{标题}]({URL}) — {一句话摘要} — *{平台}*
```

**写作要求**：
- 每个推荐必须有「为什么去」和「到了看什么/吃什么」，不许空泛
- 语气是给自己写的笔记，不是导游词
- 有确切信息写确切的，没有的不编
- 路线章节必须包含交通指引（地铁站+出口+步行时间）
- 古建不单独成章，融入路线说明中顺带介绍

### 5. 铸造便携卡片（fable-card）

从步骤 4 的 Markdown 文档中提取核心内容，铸造两张卡片，**并行执行**：

**卡片 A — 城市概览（信息图）：**

调用 Skill tool 执行 `fable-card -i`，输入内容为：城市概览 + 路线总览的精华摘要。一张图看懂这座城市怎么逛。

**卡片 B — 路线速查（长图）：**

调用 Skill tool 执行 `fable-card -l`，输入内容为路线速查专用格式。内容结构如下：

```
# {城市}路线速查

出发：{起点，如"胶东机场"} ｜ {日期范围} ｜ {天气/人流量备注}

## Day 1 · {主题名}

### 🌤️ 上午/下午 · {子主题}（约{N}h | 全程步行）

- 📍 **{地点名}** `{时间段，如10:25-10:40}` 🔶必去
  — {操作式看点：到这儿做什么、看什么细节}
  ▶ 从前一个点沿XX路向XX方向步行{N}min（经过/路过{地标}）
  
- 📍 **{地点名}** `{时间段}` ⚪可选
  — {操作式看点}
  ▶ 沿XX路向XX方向步行{N}min
  
- 🍜 **{存档咖啡}** `{时间段}` 🔶歇脚必去
  — {为什么值得停下来}
  ▶ 沿XX路向XX方向步行{N}min

🌙 晚上 · {子主题}（约{N}h）

- 🚇 **{交通换乘指引}**
  ▶ {地铁站}→{N}号线→换乘{XX}号线→{目的地站}（约{XX}min）

- 📍 **{夜市/商圈}** 🔶必去
  — {看点}

🍜 午餐/晚餐推荐：{店名} | 推荐菜 | 人均¥{XX}
```

## **Day 2 · {主题名}**

  格式同 Day 1

---

**格式要求**：
- **以天为单位组织**（Day 1 / Day 2），而非以路线编号
- 每个地点标注**时间段**（如"10:25-10:40"），让使用者知道进度
- 每条交通指引写**具体方向**（"沿XX路向XX方向步行{N}min，经过{地标}"），不仅是时长
- 每个地点标注**优先级标签**：🔶必去 / ⚪可选 / 🔶歇脚必去 / 🍺特色推荐
- **正餐点插入路线中**，而不是单独成段
- 看点必须是**操作式**的——告诉使用者「到了做什么」，而非「这是什么」
- 有**避坑信息**的地方标注 ⚠️
- 出发点灵活：开头标注起点（机场/火车站/自驾），地铁换乘指引从该起点开始计算
- 没有确切信息的地点不写入卡片

### 6. 汇总报告

```
════ 旅行研究完成 ═══════════════════════
🏙️ 城市: {城市名}
📝 知识文档: {Markdown 文件路径}
🖼️ 城市概览卡: {PNG 文件路径}
🖼️ 路线速查卡: {PNG 文件路径}
📍 路线: {N} 条 | 🏛️ 博物馆: {M} 个 | 🍜 探店: {K} 家
📎 深度内容: {X} 个视频 | {Y} 篇文章
```

## 关键约束

- 3 个 Agent 用 `parallel()` 并发，各自独立，结果不互相依赖；单个失败不影响整体
- 路线是核心产出——地理位置聚合、交通效率优先，而非按主题分类
- 古建不单独成章，融入路线中顺带介绍
- 内容权重文化侧 ~70%、潮流侧 ~30%（体现在 Agent 输出量、路线章节篇幅上）
- 没有确切信息时宁可留空，不编造
- 不产出泛泛的旅游攻略，每个推荐必须有「为什么」和「看什么」
